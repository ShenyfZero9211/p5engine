# p5engine Agent 工作规则

> 本文件定义所有 AI Agent（Kimi、Cursor、Claude 等）在参与本项目时必须遵循的工作流程与规范。

## 1. 计划管理（Plan Management）

### 1.1 强制目录
**所有制定的开发计划必须放入项目根目录的 `plans/` 文件夹中。**

- 不允许将计划文件散落在项目根目录、桌面、临时文件夹或聊天记录中。
- 如果 `plans/` 目录不存在，必须先创建它。

### 1.2 文件命名规范
计划文件必须使用以下命名格式：

```
{主题简述}_{8位随机hash}.plan.md
```

- **主题简述**：用英文小写和连字符描述计划核心内容，如 `tween-system`、`ui-theme-alpha`、`level-select-menu`
- **随机 hash**：8 位小写十六进制，用于避免文件名冲突
- **扩展名**：必须为 `.plan.md`

**示例：**
```
tween-alpha-fix_a3f7b2d1.plan.md
main-menu-animation_8e2c91f4.plan.md
```

### 1.3 计划文件内容结构
每个 `.plan.md` 必须包含以下章节：

```markdown
# 计划标题

## 目标
一句话描述本次计划要达成的目标。

## 背景
当前状态、问题描述或需求来源。

## 实施步骤
1. 具体步骤一
2. 具体步骤二
3. ...

## 验收标准
- [ ] 标准一
- [ ] 标准二

## 相关文件
- `src/main/java/.../Foo.java`
- `examples/Bar/Bar.pde`
```

### 1.4 计划生命周期
- **创建**：开始新任务前，先在 `plans/` 中创建计划文件。
- **更新**：实施过程中如有步骤变更，必须同步更新计划文件。
- **完成**：任务完成后，在计划文件顶部添加 `> **状态：已完成** ✅` 标记。
- **归档**：已完成的计划文件保留在 `plans/` 中，不要删除，作为项目历史记录。

### 1.5 执行操作
- 永远不允许自行删除任何文件，需要删除操作必须询问用户。

## 2. 删除权限（Deletion Policy）

> **绝对规则：除非用户明确批准，否则 Agent 不得自行删除任何文件，包括所有格式的文档与表格。**

### 2.1 禁止自行删除的范围
以下操作**必须**先征得用户同意（通过 `AskUserQuestion` 或直接询问）：

- 删除源代码文件（`.java`、`.pde`）
- 删除资源文件（图片、音频、字体、Shader）
- 删除配置文件（`.properties`、`.json`、`.xml`）
- 删除任何格式的文档与表格（`.md`、`.txt`、`.docx`、`.xlsx`、`.csv`、`.pdf`、`.pptx` 等）
- 删除构建产物或输出目录（`build/`、`_build_cli/`、`library/p5engine.jar`）
- 删除 Git 历史中的文件（`git rm`、`git filter-branch`）
- 删除示例项目或计划文件

### 2.2 例外情况（无需额外批准）
以下情况可在执行时附带说明，无需单独申请：

- 删除自己**刚刚创建**的临时文件（如编译缓存、截图临时文件），且已确认无其他依赖
- 清理由本次构建过程**自动生成**的临时目录（如 `build/` 下的中间 `.class` 文件），且用户已授权执行构建脚本

### 2.3 批准的形式
用户的批准必须是**明确的**，以下形式有效：

- 直接说 **"删除" / "删掉" / "可以删" / "approved"**
- 在对话中明确授权某次具体删除操作

以下**不算**有效批准：

- 沉默 / 无回应
- "你觉得呢？" / "看看行不行"
- 对代码评价（"这段没用了"）但未明确说删除

### 2.4 误删恢复
如果不小心删除了文件：
1. 立即停止后续操作
2. 通过 `git checkout -- <file>` 或备份恢复
3. 向用户汇报误删及恢复情况

## 3. 代码规范

### 3.1 最小修改原则
- 只修改与当前任务直接相关的代码。
- 不要顺便格式化、重构或"优化"无关文件。

### 3.2 编译验证
- 每次修改 Java 源码后，必须运行 `compile-jar.ps1` 重新编译 `p5engine.jar`。
- 每次修改示例 PDE 后，必须用 Processing CLI (`processing.exe cli --sketch=... --build`) 验证编译通过。
- **禁止将未编译验证的代码提交给用户。**

### 3.3 PDE 语法注意
- Processing 的 PDE 预处理器不支持 Java 静态字段作为方法引用。
- **正确**：`Ease::outQuad`
- **错误**：`Ease.outQuad`（会被预处理器错误解析）

## 4. 文件与目录操作

### 4.1 工作目录边界
- 默认工作目录为项目根目录 `E:\projects\kilo\p5engine`。
- 所有文件操作（创建、修改、删除）默认在此目录内进行。
- **禁止**访问或修改用户桌面、`C:\Users\...\Documents` 等个人目录，除非用户明确要求。

### 4.2 输出目录
- 引擎构建输出：`library\p5engine.jar`
- 如果此时需要编译示例pde程序：把`library\p5engine.jar`复制到对应pde示例工程根目录中的`code`文件夹中
- CLI 示例构建输出：`{sketch}\_build_cli\` 或 `{sketch}\build\`
- 通常不需要执行`processing.exe cli --sketch=... --build --output...`导出exe指令，不需要导出独立应用，除非用户需要或命令
- 计划文件：`plans\*.plan.md`

## 5. 沟通与汇报

### 5.1 截图验证
- UI/动画类修改完成后，运行后如果用户想要截图验证视觉效果，那么执行截图，否则不执行。
- 截图保存到示例目录或临时目录，供用户确认。

### 5.2 总结文档
- 涉及架构调整、Bug 修复或功能新增的任务完成后，应在 `docs/` 目录下创建简短的 Markdown 总结文档。
- 文档应包含：问题描述、根本原因、修复内容、验证结果。

## 6. 版本与兼容性

- **Processing 版本**：4.5.2
- **JDK 版本**：17
- **目标平台**：Windows（开发环境）
- 引入新依赖前必须征得用户同意。

## 7. Git 操作规范

> **绝对规则：除非用户明确给出 commit / tag / push 相关指令，否则 Agent 不得执行任何 Git 变更操作。**

### 7.1 禁止自动执行
以下操作**绝不自动执行**，即使任务已完成或用户说"可以了"、"不错"：

- `git commit`
- `git tag`
- `git push`
- `git reset` / `git rebase` / `git revert`

### 7.2 有效指令形式
用户必须**明确说出**以下动词才算授权：

- **"提交" / "commit"**
- **"推送" / "push"**
- **"打标签" / "tag"**
- **"发布" / "release"**

### 7.3 无效形式（不能执行）
以下**不算**授权：

- "看起来可以了"
- "不错"
- "保存一下"（指保存文件，不是 git commit）
- 沉默或无回应

### 7.4 例外
用户明确说 **"每次需要 git 操作时都先问我"** 时，即使听到了 commit 相关词汇，也要先确认再执行。
